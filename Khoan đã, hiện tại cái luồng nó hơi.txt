## 🏗️ PROJECT CONTEXT

GPMS centralizes the entire graduation project lifecycle: topic approval, group management, supervisor/reviewer assignment, checklist-based evaluation, feedback approval workflow, and defense scheduling.

**Tech stack:** ASP.NET Core MVC + Razor Views + Bootstrap 5  
**Design target:** Professional academic enterprise system. NOT a casual student app. Clean, structured, information-dense but readable.

---

## 👤 USER ROLES (Lecturer)

A single lecturer account can hold both roles simultaneously across different projects:

**Mentor (Supervisor):**  
Supports assigned student groups throughout the project. After a Reviewer submits evaluation feedback for a group, the Mentor is responsible for reviewing that feedback, then deciding to Approve, Reject, or Edit it before it reaches the students.

**Reviewer:**  
Assigned by the Head of Department to review specific groups in specific rounds. The Reviewer evaluates the group's submitted documents using a checklist (scored criteria), then writes and submits feedback.

## LECTURER ROLES — BUSINESS LOGIC

In GPMS, a single Lecturer account can hold both roles simultaneously across different projects. The system must reflect both roles in the same portal.

### Mentor (Supervisor)
- Assigned to a project by the Head of Department (HOD)
- Responsible for supporting and tracking their assigned student groups throughout the entire project lifecycle
- When a Reviewer submits an evaluation + feedback for a group:
  - The system notifies the Mentor
  - The Mentor must review the feedback within a configured time window
  - The Mentor can: Approve (send as-is to students), Edit then Approve (modify content before sending), or Reject (block it from reaching students)
  - If the Mentor does NOT act within the deadline → system AUTO-RELEASES the feedback directly to students (ApprovalStatus = "AutoReleased")
- A Mentor CANNOT be assigned as Reviewer for any group they are supervising (enforced by system constraint)

### Reviewer
- Assigned to specific groups per review round by the HOD (auto-assigned with conflict-avoidance constraints)
- Constraints enforced by system:
  - Cannot review a group they are supervising
  - Cannot review the same group in more than one round
  - Cannot exceed the maximum number of assigned groups per round
- Per assigned group per round, the Reviewer must:
  1. Access the group's submitted documents
  2. Score each checklist criterion (each has a max score, configured by HOD)
  3. Write an overall feedback comment
  4. Submit the evaluation → triggers Mentor notification
- There are 3 review rounds total:
  - Round 1 (Online) — Google Meet link provided
  - Round 2 (Online) — Google Meet link provided
  - Round 3 / Defense (Offline) — physical room assigned
- Once submitted, the Reviewer cannot edit the evaluation unless explicitly allowed

### Key workflow between the two roles
Reviewer submits evaluation
→ Feedback created with status "Pending"
→ Mentor receives notification
→ Mentor approves / edits / rejects within deadline
→ If approved: students can see feedback (IsVisibleToStudent = true)
→ If auto-released: students see feedback, Mentor is notified of auto-release
→ If rejected: feedback is blocked, Reviewer may be notified
